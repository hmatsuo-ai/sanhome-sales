# 作業時の必須確認（Remain）

**Cursor / AI にプロンプトを渡すたび、またはコード・UI を変更するたびに、次の3点を確認すること。**

人間がレビューする場合も同様にチェックする。

---

## 1. 仕様書どおりか

変更・追加した内容が、以下の仕様と**矛盾しないか**・**意図した範囲に収まっているか**を確認する。

| 優先度 | 資料 | 内容の目安 |
|--------|------|------------|
| 高 | [`SPEC.md`](SPEC.md) | 機能・データ・API・権限・運用の一次仕様 |
| 高 | [`docs/やさしい仕様書.md`](docs/やさしい仕様書.md) | 利用者向けに「できること」の説明との整合 |
| 中 | [`docs/ユーザー向け仕様書.md`](docs/ユーザー向け仕様書.md) | ユーザー向け説明との整合 |
| 参照 | [`spec/screens.md`](spec/screens.md) | 画面仕様 |
| 参照 | [`spec/api.md`](spec/api.md) | API 仕様 |
| 参照 | [`spec/db.md`](spec/db.md) | DB 仕様 |
| 参照 | [`spec/gaps.md`](spec/gaps.md) | 既知ギャップ（仕様未決・差分） |

- 仕様に**ない**機能を勝手に広げていないか。
- 仕様で**決まっている**権限・判定・項目名を変えていないか（変えるなら仕様書の更新もセットで検討する）。

---

## 2. 誰でも使いやすく・シンプルな UI か

次を満たすよう意識する。迷ったら**削る・統一する・文言を短くする**。

- **一画面の目的が一つに見える**（関係ない操作を並べない）。
- **主要操作が目立つ**（ボタン階層・ラベルが分かりやすい）。
- **専門用語を減らす**（仕様上必要な用語はツールチップや補足一文で）。
- **既存のデザインに合わせる**（`card` / `btn` / `form-input`、余白・タイポの流儀を崩さない）。
- **営業・管理者の両方**が迷わず触れるか（権限が違っても「何ができないか」が分かる表示になっているか）。

大きな UI 変更をしたら、**触る人が IT に不慣れでも説明なしで一通り試せるか**を想定して見直す。

---

## 3. Git へプッシュしたか（変更を入れたら毎回）

コード・設定・仕様書・本ファイルを**実質的に変更したら**、次を行う。

1. `git status` で意図しない差分がないか確認する。
2. コミットメッセージは**何をしたかが一文で分かる**ように書く。
3. **`main`（または運用ブランチ）をリモートにプッシュする**までを完了とみなす。  
   - プッシュできない環境なら、**不足している作業（プッシュ未実施）を明示**する。

「ローカルだけ更新して終わり」にしない。
